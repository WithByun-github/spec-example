# PRD — K-LLM Firewall

> 상세 기획 / 요구사항 정의서. "무엇을, 왜, 어떤 기준으로 만드는가"를 정의한다.
> Claude Code가 기능을 구현할 때 판단 기준으로 삼는 문서.

---

## 1. 배경 (왜 만드는가)

- ChatGPT 등장 이후 전 산업에서 AI 챗봇·Agent 도입이 가속화되며 **새로운 보안 위협**이 등장.
- AI는 "대화로 속이는 공격(프롬프트 인젝션·탈옥)"이 가능 → 기존 방화벽·백신으로는 막을 수 없는 신영역.
- 한 줄 입력으로 기밀이 새는 사고 발생 (예: 코드 유출, 챗봇 오답 사고).
- 한국엔 검증된 AI 보안 솔루션 부재. 외산(Lakera 등)은 영어 중심 → 한국어 공격·한국형 개인정보 대응 한계.
- 동시에 Byun그룹은 보안솔루션을 외산·외주에 의존 → 비용 누적 + 전략적 종속 리스크.

**→ 이 두 흐름의 교차점이 LLM 방화벽.** AI Agent로 자체 구축하여 "내재화"의 첫 Pilot으로 삼는다.

이때 K-LLM Firewall은 단독 솔루션이 아니라, **내재화한 보안솔루션들을 한 곳에서 통합 관리하는 "내재화 플랫폼"의 첫 번째 입주 솔루션**으로 만든다. 이후 로그분석·취약점 트리아지·컴플라이언스 점검 등 후속 솔루션이 같은 플랫폼 위에 쌓인다.

## 2. 목표

### 2.1 제품 목표
- 회사가 쓰는 AI 챗봇(주로 외부 SaaS LLM) 앞단에 두어, **위험 입력 차단 + 민감정보 유출 방지**를 코드 수정 없이 적용. → 사내 데이터가 SaaS로 새어 나가는 것을 막는 것이 핵심.
- 한국어 공격 패턴 / 한국형 개인정보(주민번호·카드·사업자번호 등)에 특화.
- 내재화 플랫폼의 첫 모듈로 동작하며, 탐지 현황·관리 설정을 플랫폼 화면에서 제공.

### 2.2 Pilot(증명) 목표
- "AI Agent(바이브 코딩)로 고급 보안 개발자 공백을 대체해 보안솔루션을 자체 구축할 수 있다"는 가설 실증.
- 검증된 개발·검증·운영 파이프라인을 후속 보안시스템(로그분석·취약점 트리아지 등)에 재사용.

## 3. 사용자 / 이해관계자

| 역할 | 니즈 |
| --- | --- |
| 챗봇 사용자 | 평소엔 게이트웨이 존재를 못 느낄 만큼 투명. 위험 시에만 차단 안내. |
| 보안 담당자 | 어떤 공격이 언제 들어왔는지 플랫폼 화면으로 확인, 알림 수신, 룰 관리. |
| 챗봇 개발팀 | 기존 챗봇 코드 수정 없이 게이트웨이만 앞에 두면 됨 (드롭인). |
| 경영진 | "비용센터 → 수익센터 전환", 외산 의존 탈피의 증거. 플랫폼으로 내재화 진척을 한눈에. |

## 4. 핵심 기능 요구사항

### F1. 입력 필터 (Input Guardrail)
- **F1-1** 프롬프트 인젝션 탐지 (예: "이전 지시 무시하고...", "시스템 프롬프트 알려줘")
- **F1-2** 탈옥(Jailbreak) 탐지 (역할극 우회, DAN류 등)
- **F1-3** 인코딩 우회 탐지: Base64 등으로 변형된 입력을 원문 복원 후 재검사
- **F1-4** 입력 민감정보 사전 마스킹: 사용자가 실수로 넣은 개인정보·기밀이 LLM에 도달하기 전 차단/마스킹

### F2. 출력 필터 (Output Guardrail)
- **F2-1** RAG·도구 호출로 주입된 사내 데이터가 응답에 노출되는 케이스 차단
- **F2-2** 다른 세션 사용자 입력이 응답에 반향되는 케이스 차단
- **F2-3** 권한 없는 사용자에게 민감정보 도달 전 마스킹

### F3. 정책 엔진 (Policy Engine)
- **F3-1** 챗봇 역할 이탈 탐지 (정의된 역할/도메인 벗어나는 요청)
- **F3-2** Off-topic 탐지
- **F3-3** 컴플라이언스 위반 탐지

### F4. 2단계 검사
- **F4-1** 1차(룰): 키워드·패턴 사전 매칭으로 명백한 공격 즉시 차단 (목표 <50ms)
- **F4-2** 2차(로컬 LLM): 1차 애매한 경우만 Ollama 호출하여 자연어 의도 분석
- **F4-3** 2차는 **마스킹 전 원문**을 다루므로 외부로 보낼 수 없음 → 사내 로컬(Ollama)만 사용. (외부로 보내면 방화벽이 막으려던 유출을 스스로 저지르는 셈)

### F5. 한국형 PII 탐지 (입출력 양방향)
- 주민등록번호, 휴대폰번호, 카드번호, 사업자등록번호, 계좌번호, 여권번호 등 한국 특화 패턴.
- 입력단: LLM 도달 전 마스킹. 출력단: 사용자 노출 전 마스킹.
- 검증 로직 포함 (예: 주민번호 체크섬으로 오탐 감소).

### F6. 로그·시각화·알림
- **F6-1** 모든 검사 결과를 SQLite에 기록 (타임스탬프, 판정, 사유, 검사단계, 처리시간, 마스킹된 입력 일부)
- **F6-2** 플랫폼 내 Firewall 모듈(Streamlit): 탐지 추이, 공격 유형 분포, 차단/통과 비율, 최근 탐지 목록
- **F6-3** 위협 탐지 시 보안팀 알림 (해커톤은 로그/콘솔/웹훅 stub으로 충분)

### F7. AI Agent 활용 (구축 자동화 — 가설 실증의 핵심)
- **F7-1** 공격 사례 수집 Agent: 보안 커뮤니티·논문에서 최신 공격 패턴 수집·정리
- **F7-2** 탐지 룰 작성 Agent: 신종 공격 발견 시 룰 초안 자동 작성 → 담당자 검토·승인
- **F7-3** 자동 테스트 Agent: 공격 시나리오 자동 생성 → 탐지율·오탐율 자동 리포팅

### F8. 내재화 플랫폼 (통합 관리 화면)
- **F8-1** 솔루션 목록/통합 현황: 내재화된 솔루션 카드 (K-LLM Firewall=실제, 후속 솔루션=목업)
- **F8-2** K-LLM Firewall 모듈: F6-2의 탐지 대시보드를 플랫폼의 한 탭으로 제공
- **F8-3** 관리 설정: 검사 동작(2차 on/off, fail-safe), 룰 관리, 정책, 알림 채널, LLM 설정(보호 대상·2차 검사 로컬)
- **F8-4** 개발 현황: 각 솔루션 구축 단계·진행률 + AI Agent 산출물 가시화
- **F8-5** 해커톤 구현 수준: Firewall 관련 모듈은 실데이터 연동(실제), 나머지는 목업으로 비전 전달

## 5. 비기능 요구사항

- **호환성**: OpenAI Chat Completions API 호환 (`/v1/chat/completions`) → 기존 챗봇이 base_url만 바꾸면 적용.
- **성능**: 1차 검사로 대다수 통과 트래픽은 저지연. 2차는 필요 시에만.
- **장애 안전(fail-safe)**: 검사기 오류 시 기본 차단 또는 명시적 정책. config로 결정.
- **데이터 주권**: 마스킹 전 원문은 외부로 전송하지 않음 (2차 검사는 로컬 전용). 보호 대상 SaaS 챗봇에는 마스킹·검사를 통과한 입력만 전달.
- **확장성**: 새 공격 유형/룰을 코드 수정 없이 yaml 추가로 확장.

## 6. 성공 기준 (해커톤 데모 기준)

- [ ] 알려진 인젝션/탈옥 공격 샘플을 1차에서 즉시 차단하는 것을 시연
- [ ] Base64 우회 공격을 복원·차단 시연
- [ ] 주민번호 등 한국형 PII가 입력·출력 양방향에서 마스킹되는 것을 시연
- [ ] 애매한 입력이 2차 로컬 LLM으로 넘어가 판정되는 흐름 시연
- [ ] 플랫폼 화면에서 탐지 이력·통계 확인, 다른 솔루션은 목업으로 비전 제시
- [ ] "이 전부를 AI 코딩으로 며칠 만에 만들었다" = 내재화 가설 실증 메시지 전달

## 7. 범위 밖 (해커톤에서 안 함)

- 실시간 고가용성 클러스터링, 멀티테넌시
- 정교한 RBAC (권한 모델은 단순화)
- 실제 사내 인증 시스템 연동 (mock으로 대체)
- 후속 보안시스템(로그분석·취약점 관리)의 실제 구현 (방향만 제시)

## 8. 열린 질문 (추후 확정)

- 보호 대상 챗봇(SaaS 또는 사내)의 실제 엔드포인트 URL / 인증 방식 → 데모에선 mock 또는 Ollama로 대체 가능.
- 2차 검사용 Ollama 모델명 (한국어 성능 좋은 모델) → 환경에 맞게 `.env`로 설정.
- 알림 채널 (이메일/슬랙/사내 메신저) → 데모는 웹훅 stub.
