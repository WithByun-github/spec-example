| 차원 | 확인된 사실 | 영향 |

| 팀 구성 | 3명 | — |

## 2. 종합평 (통합)

🟡 **보안 설계는 수준급으로 잘 짜였고(1차), 팀 역량·환경도 충분하다(2차). 유일한 적은 "넓은 범위 ÷ 하루 1~2시간·비동기"다 — 핵심으로 좁히면 완주 가능.**

- **설계 측면(1차):** "2차 LLM은 외부로 보내지 않는다"는 원칙은 이 분야를 제대로 이해한 팀이라는 신호. 실연동/목업 분리도 현명함.
- **실행 측면(2차):** 치명적 환경 제약이 없고, 핵심(F1)=데모 하이라이트(실시간 차단)=가장 적은 시간으로 되는 슬라이스가 완벽히 정렬돼 있음.
- **공통 처방:** Firewall 한 줄기를 데모 척추로 고정하고, 나머지는 스트레치/목업으로 밀 것.

## 3. 잘한 점 (통합 — 살릴 강점)

### A. 보안 설계의 정합성 *(1차)*
- **2차 검사 LLM 외부 전송 금지** — "마스킹 전 원문 의도 분석을 외부에 보내면 방화벽이 막으려던 유출을 스스로 저지르는 셈"이라는 통찰. 방어 도구의 자기모순을 미리 차단.
- **2단계 검사 설계** — 1차 룰(빠름) / 2차 로컬 LLM(정확)으로 속도·비용·정확도 트레이드오프를 잘 잡음.
- **룰을 코드에 박지 않음** — `rules/*.yaml`에서 로드해 확장 가능.
- **사업화 디테일** — MIT/Apache 2.0 라이선스, fail-safe(실패 시 차단) 같은 운영 안전장치까지 고려.

### B. 구현 범위의 현명한 분리 *(1차·2차 공통)*
- K-LLM Firewall = 실연동(SQLite 실데이터), 나머지 플랫폼 = 목업. **"핵심 하나는 실연동, 확장 비전은 목업"** → 완성도와 스토리를 동시에 잡음.

### C. 스펙 완성도 *(2차)*
- WHAT·WHY·범위 밖·성공 기준·데모 시나리오·예상 질문까지 닫혀 있음. 해커톤 스펙이 이 정도로 정리된 경우는 드묾. *(1차 시점의 "문제·사용자·성공기준 공백"이 보완된 결과로 보임 — 4-A)*

### D. 탄탄한 기반과 진행 원칙 *(2차)*
- 실서비스 백엔드 경험 + AI코딩 완주 경험 + Ollama 구동 + 외부 API 자유 → 작품을 막을 **치명적 환경 제약 없음.**
- "패스스루부터 만들고 항상 돌아가는 상태 유지" + Phase별 검증 게이트 → 비동기·시간 분할 팀에 최적.

## 4. 개선할 점 (통합 — 완주를 위협하는 것)

> 1차와 2차가 **같은 곳을 가리키는 항목이 많다.** 중복은 합치고, 우선순위 순으로 정렬했다.

### ① 범위 과다 — 데모 척추를 Firewall 한 줄기로 고정 *(1차·2차 공통, 최우선)*
- **2차:** 실작업 ~90시간에 F1~F8 + 8 Phase + 플랫폼 셸 + AI Agent 자동화(F7)가 경쟁. 특히 **출력 필터(F2)·정책 엔진(F3)·AI Agent 자동화(F7)**가 핵심(F1)과 동급 무게로 시간을 빨아들임.
- **1차:** Firewall 자체만으로도 충분히 큰데 플랫폼 셸까지 겹침. 플랫폼은 목업이니 **최소 틀**로만.
- **합친 처방:** 아래 한 흐름을 핵심으로 고정 →
  > 공격 프롬프트·민감정보 입력 → 1차 룰 차단 또는 2차 판정 → 마스킹된 안전 입력만 외부 LLM 전달 → 탐지 로그·대시보드 표시

### ② 우선순위 / 명시적 "제외 범위" 부재 *(1차·2차 공통)*
- **2차:** 성공 기준 6개가 전부 동등하게 적혀 "무엇을 버릴지"가 문서에 없음 → 막판 흔들림 위험.
- **1차:** 7절 "절대 하지 말 것"은 보안 규칙이지 "이번 MVP에서 안 만들 기능"이 아님. 후속 솔루션(로그분석·취약점 트리아지·컴플라이언스)이 목업이라는 점을 **제외 범위로 못박을 것.**
- **합친 처방:** 스펙에 **우선순위 + '안 할 것' 한 줄**을 명시적으로 박아 넣기.

### ③ 2차 검사 로컬 LLM(Ollama)이 시연 변수 *(1차)*
- 환경 셋업·응답 속도가 발표 당일 변수. 이미 설계에 있는 **fail-safe를 데모 안정성으로 활용** — 2차 LLM이 불안정해도 **1차 룰 차단만으로 핵심 시연이 끊기지 않게** 구성하고, 2차는 소수 시연 케이스로 좁힐 것.

### ④ 시간 구조가 숨은 변수 *(2차)*
- 하루 1~2시간 + 비동기는 캘린더 3주보다 훨씬 짧게 작동. 매 세션 컨텍스트 재적재·조율 비용으로 실효 시간이 더 깎임. → ①②의 범위 축소가 그만큼 더 중요.

### A. (참고) 1차의 "문제·사용자·성공 기준 공백"은 보완된 것으로 보임
- 1차 피드백은 제출물이 spec이 아닌 `CLAUDE.md`라 "누가 어떤 상황에서 불편한지"와 "3주 시연에서 뭘 보여줄지"가 비어 있다고 지적했음.
- 2차 피드백 시점에는 "성공 기준·데모 시나리오까지 닫혀 있다"로 평가됨 → **이미 해소.** 만약 아직 사용자 한 명의 구체 상황(예: "직원이 사내 챗봇에 고객 정보를 붙여넣을 때")이 약하다면 그 한 문장만 더 보강.
