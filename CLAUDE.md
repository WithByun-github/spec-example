## 1. 프로젝트 한 줄 요약

**보안솔루션 내재화를 위해 "보안솔루션 통합 관리 플랫폼"을 구축하고, 그 첫 번째 솔루션으로 LLM 방화벽(K-LLM Firewall)을 개발한다.**

- **내재화 플랫폼(Security Internalization Platform)**: 그룹이 자체 구축한 보안솔루션들을 한 곳에서 통합 관리·모니터링하는 콘솔. 이후 로그분석·취약점 트리아지·컴플라이언스 점검 등 후속 솔루션이 같은 플랫폼 위에 차례로 쌓인다. (상세는 2-1 참조)
- **K-LLM Firewall (첫 번째 솔루션)**: AI 챗봇 앞단에 두는 한국어·한국 도메인 특화 "보안 검문소(Security Gateway)". 사용자와 회사가 쓰는 LLM 챗봇 사이에서 **위험한 입력을 차단**하고 **민감정보 유출을 막는다.**

이 모든 것을 **AI Agent(바이브 코딩)로 자체 구축**한다 — "AI Agent로 보안솔루션을 자체 구축할 수 있다"는 것이 이 프로젝트로 증명하려는 핵심이다.

## 2. 무엇을 만드는가 (해커톤 범위)

해커톤에서 만드는 것은 **플랫폼의 첫 번째 솔루션인 K-LLM Firewall(실제 동작)** 과, 그것을 품는 **내재화 플랫폼 셸(일부 실제·일부 목업, 2-1 참조)** 이다.
아래는 그중 핵심인 K-LLM Firewall의 동작이다.

사용자와 회사가 쓰는 챗봇(1개) 사이에 끼우는 **API 게이트웨이**. 처리 흐름:

```
[사용자 입력]
   → ① 입력 검사 (공격 프롬프트 차단 + 사용자가 보낸 민감정보 마스킹)
   → ② 챗봇 LLM 호출 (마스킹된 안전한 입력만 전달)
   → ③ 출력 검사 (LLM 응답 내 민감정보·기밀 마스킹)
[최종 응답]
```

> 핵심: 보호 대상 챗봇은 대개 **외부 SaaS LLM(ChatGPT 등)** 이다. 사내 데이터가 그 SaaS로 새어 나가는 것을 막는 것이 이 방화벽의 목적이다. 따라서 ②로 나가는 입력은 항상 ①에서 마스킹·검사를 통과한 안전한 입력이다.
> 반면 **2차 검사용 LLM은 외부로 보낼 수 없다** — 2차 검사는 마스킹 전 원문의 의도를 분석하므로, 외부에 보내면 방화벽이 막으려던 유출을 스스로 저지르는 셈이 된다. 따라서 2차 검사 LLM은 사내 로컬(Ollama)로만 운용한다.

핵심 기능 3가지:
1. **입력 필터(Input Guardrail)**: 프롬프트 인젝션 / 탈옥 / 인코딩 우회 탐지·차단 + 입력 민감정보 사전 마스킹
2. **출력 필터(Output Guardrail)**: 응답에 섞인 민감정보·사내 기밀 마스킹
3. **정책 엔진(Policy Engine)**: 챗봇 역할 이탈·Off-topic·컴플라이언스 위반 탐지

검사는 **2단계**:
- **1차(룰 기반)**: 알려진 공격 키워드·패턴 사전 매칭 → 명백한 공격 즉시 차단 (빠름)
- **2차(로컬 LLM)**: 1차에서 애매한 경우만 Ollama 로컬 LLM으로 의도 정밀 분석 (정확)

모든 탐지는 **로그 저장 + 플랫폼 내 시각화 + 보안팀 알림**.

### 2-1. 내재화 플랫폼 (Security Internalization Platform)

K-LLM Firewall을 품는 **상위 통합 관리 페이지**다. 내재화한 보안솔루션들을 한 곳에서 통합 관리·모니터링하며, K-LLM Firewall이 그 첫 번째 입주 솔루션이다.

플랫폼은 다음을 한 화면 체계 안에서 제공한다 (좌측 내비게이션 + 모듈별 페이지):
- **솔루션 목록 / 통합 현황**: 내재화된 솔루션들을 카드로 나열 (K-LLM Firewall = 실제, 나머지 = 목업: 로그분석·취약점 트리아지·컴플라이언스 점검 등 후속 솔루션 자리)
- **K-LLM Firewall 모듈**: 탐지 대시보드를 **플랫폼의 한 탭/모듈로 제공** (탐지 추이·유형 분포·차단율·최근 탐지)
- **관리 설정**: 검사 동작(2차 검사 on/off, fail-safe), 룰 관리(yaml 목록·승인 상태), 정책 설정, 알림 채널, LLM 설정(보호 대상 LLM·2차 검사 로컬 LLM) 등
- **개발 현황(내재화 진척)**: 각 솔루션의 구축 단계·진행률, AI Agent 자동화 산출물(공격 수집·룰 자동생성·자동 테스트 리포트) 가시화 → "AI Agent로 자체 구축 중"이라는 가설을 시각적으로 증명

**해커톤 구현 수준**:
- K-LLM Firewall 관련 모듈(탐지 대시보드, Firewall 설정)은 **실제 동작** (SQLite 실데이터 연동).
- 나머지 플랫폼 영역(다른 솔루션·일부 개발 현황)은 **목업**으로 비전을 전달.
- 즉, **"단일 솔루션이 아니라 내재화 플랫폼의 첫 조각"** 이라는 스토리를 화면으로 보여주는 것이 목적.

> 구현 메모: 탐지 대시보드는 독립 화면이 아니라 **플랫폼 셸 안의 한 모듈(`platform/modules/firewall.py`)** 로 만든다. 플랫폼 셸(`platform/app.py`)은 좌측 내비 + 페이지 라우팅 구조로, Firewall 모듈만 실데이터에 연결하고 나머지는 목업 컴포넌트로 둔다.

## 3. 기술 스택

| 구분 | 기술 | 용도 |
| --- | --- | --- |
| 언어 | Python 3.11+ | 전체 |
| 게이트웨이 | FastAPI + Uvicorn | 보안 검사 관문 (OpenAI 호환 API 프록시) |
| 가드레일 | LLM Guard | 프롬프트 인젝션·탈옥 탐지 룰 참고·활용 |
| PII 탐지 | Microsoft Presidio | 글로벌 PII 엔진 (+ 한국형 패턴 자체 추가) |
| LLM 연동 | LiteLLM | 보호 대상 LLM 통합 호출 (SaaS LLM 기본, 사내 로컬 LLM도 지원) |
| 2차 검사 LLM | Ollama (로컬 전용) | 마스킹 전 원문 의도 분석 — 외부 전송 금지 |
| 저장소 | SQLite | 탐지 이력 로그 |
| 대시보드/플랫폼 | Streamlit | 내재화 플랫폼 셸 + Firewall 탐지 시각화 |

**라이선스**: 모두 MIT / Apache 2.0 (상업적 활용 가능) → 향후 사업화 시 라이선스 부담 없음.

## 5. 코딩 규칙 (Claude Code가 지킬 것)

- **언어**: 코드 주석·docstring·로그 메시지는 한국어 OK. 변수/함수명은 영어.
- **검사 로직은 모듈로 분리**: 새 공격 유형이 생기면 `guards/`에 함수만 추가하면 되도록 확장 가능하게.
- **룰은 코드에 하드코딩하지 말 것**: 공격 패턴은 `rules/*.yaml`에서 로드. (AI Agent가 룰 초안을 자동 생성·갱신하는 컨셉)
- **모든 차단/통과 결정은 로그로 남길 것**: 입력 일부(마스킹된 형태), 판정 결과, 사유, 검사 단계(1차/2차), 처리시간.
- **민감정보는 로그에도 평문 저장 금지**: 마스킹된 형태로만 저장.
- **에러는 삼키지 말 것**: 검사기가 실패하면 기본은 "차단(fail-safe)" 또는 명시적 통과 정책을 config로 결정.
- **마스킹 전 원문은 외부로 내보내지 말 것**: 2차 검사는 사내 로컬 LLM(Ollama)만 사용한다. (보호 대상 챗봇이 외부 SaaS여도, 그쪽으로는 ①에서 마스킹·검사를 통과한 안전한 입력만 전달)

## 6. 작업 시작 방법

새 작업을 시작하기 전:
1. `docs/TASKS.md`에서 현재 단계 확인
2. 관련 있으면 `docs/PRD.md` / `docs/ARCHITECTURE.md` 참조
3. 완료하면 `docs/TASKS.md` 체크박스 갱신

## 7. 절대 하지 말 것

- 마스킹 전 원문(사용자 입력 원본)을 외부 API로 전송하지 않는다. 특히 2차 검사는 반드시 사내 로컬 LLM으로만 수행한다. (보호 대상 SaaS 챗봇 호출은 예외 — 단, 마스킹·검사를 통과한 입력만 전달)
- 검사 룰을 임의로 약화시키지 않는다 (오탐 줄이려고 탐지 끄기 금지 — config로만 조정).
- 데모용 가짜 데이터에 실제 개인정보를 쓰지 않는다 (항상 합성 데이터).
